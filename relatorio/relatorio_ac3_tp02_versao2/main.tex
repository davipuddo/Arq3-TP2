\begin{abstract}
\textbf{Abstract.~}
\noindent This report describes the implementation of key components in a five-stage pipeline data path for a simplified RISC-V processor, based on the Patterson and Hennessy architecture. While pipelining improves CPU performance through instruction interleaving, it introduces structural, data, and control hazards. This work addresses these challenges by implementing flow control mechanisms, specifically the branch-equal (BEQ) instruction, and developing logic for hazard detection and forwarding. The development was divided into three main phases: the initial implementation of branch instructions and control flow; the execution and manual correction of assembly test programs using NOPs and instruction reordering; and the final integration of a Hazard Detection Unit and Forwarding Unit to handle load-use dependencies and ALU data hazards automatically. The results demonstrate the efficiency of these hardware-level solutions in maintaining execution integrity without the need for manual code intervention. 
\end{abstract}

\begin{abstract}
\textbf{Resumo.~}
\noindent Este relatório descreve a implementação de componentes chave em um caminho de dados com pipeline de cinco estágios para um processador RISC-V simplificado, baseado na arquitetura de Patterson e Hennessy. Embora o pipelining melhore a performance da CPU através da intercalação de instruções, ele introduz dependências estruturais, de dados e de controle. Esse trabalho aborda tais desafios mediante a implementação de mecanismos de controle de fluxo, especificamente a instrução branch-equal (BEQ), e o desenvolvimento de lógica para detecção de dependências e forwarding. O desenvolvimento foi dividido em três principais fases: a implementação inicial das instruções de desvio e controle de fluxo; a execução e correção manual de programas de teste em assembly utilizando NOPs e reordenação de instruções; e a integração final de uma Unidade de Detecção de Dependências e Unidade de Forwarding para lidar com dependências do tipo load-use e dependências de dados na ULA automaticamente. Os resultados demonstram a eficiência destas soluções a nível de hardware em manter a integridade da execução sem a necessidade de intervenção manual.
\end{abstract}

\section{Introdução}
\label{sec:intro}

Algumas arquiteturas de processadores têm como característica a execução sequencial das instruções. Este tipo de execução é ineficiente, uma vez que a própria arquitetura limita o número de instruções que podem ser concluídas por unidade de tempo, configurando uma subutilização dos recursos. De modo a aumentar a utilização da CPU e consequentemente processar mais instruções por unidade de tempo, a técnica de pipelining surge para dar capacidade ao processador de sobrepor múltiplas instruções em execução.

Apesar de ser poderosa, a sobreposição de instruções cria relações de dependência (hazards) que devem ser tratadas assertivamente para garantir a execução correta. Os três tipos de hazard são:
\begin{itemize}
    \item \textbf{Dependência Estrutural (Structural Hazard)}: múltiplas instruções necessitam de um mesmo módulo, como a memória;
    \item \textbf{Dependência de Dados (Data Hazard)}: múltiplas instruções necessitam de um mesmo registrador;
    \item \textbf{Dependência de Controle (Control Hazard)}: a instrução atual é um desvio condicional, mas devido ao pipeline a instrução localizada no próximo endereço começa a ser executada antes do processamento da condição.
\end{itemize}

Enquanto a própria arquitetura é capaz de resolver o problema da dependência estrutural por meio da multiplicação de seus módulos, as demais dependências necessitam de intervenções na ordem das instruções e do pipeline. 

O objetivo deste trabalho é implementar partes do caminho de dados em um processador RISC-V simplificado com pipeline de cinco estágios. O modelo de processador utilizado é apresentado no livro Computer Organization and Design Risc-V Edition. \cite{patterson2017organizacao}.

O RISC-V é uma ISA (Instruction Set Architecture) universal de tipo RISC (Reduced Instruction Set Computing) cuja principal característica é a modularidade. O RISC-V possui instruções básicas capazes de executar uma pilha completa de software de forma independente. Dizer que um processador RISC-V possui um pipeline de cinco estágios significa, portanto, que ele: (1) segue um ISA atômico e modular; (2) utiliza uma técnica de paralelismo em seu caminho de dados que permite que a execução de até cinco instruções se sobreponha simultaneamente no tempo para aumentar o rendimento e melhorar o desempenho. \citep{patterson2017riscv}

Dessa forma, neste trabalho adicionaremos funcionalidades ao pipeline para branch e controle de fluxo e hazard detection e forwarding, bem como executar um programa de teste para detecção de dependências e correção.

\section{Desenvolvimento}

\subsection{Branch e Controle de Fluxo}

O controle de fluxo pode ser definido como os mecanismos de instruções de saltos e desvios, que alteram a ordem de execução sequencial de um programa. Nesse contexto, um branch é uma instrução de desvio condicional que faz o processador saltar para o endereço de destino apenas se a comparação entre dois registradores for verdadeira. Caso a condição imposta não seja atendida, o processador executa a próxima instrução sequencial. \cite{patterson2017riscv}

Nesta etapa, implementamos o desvio condicional e tratamos hazards de controle. Para isso, implementamos: a instrução BEQ, a comparação de  operandos no estágio EX, o cálculo endereço de destino do branch e a atualização correta do Program Counter (PC).

Dessa forma, quando um branch é tomado, o PC passa a apontar para o endereço correto e as instruções já carregadas são descartadas.

\subsection{Primeira Execução do Programa de Teste}

Até este ponto, o caminho de dados do RISC-V ainda não possui mecanismos para a detecção de dependências e correção. O programa de teste em questão possui dependências e, portanto, para garantir a execução sem erros, foi necessário reordenar as instruções manualmente e inserir instruções de \texttt{NOP} em alguns locais, além de corrigir manualmente o endereço da label para o desvio da instrução \texttt{BEQ}.

No entanto, nosso objetivo era garantir a execução correta do programa sem a necessidade de alterações manuais ou inserção de \texttt{NOP}s no programa. Para tal fim, implementamos dois módulos adicionais para o projeto de CPU: Um para a detecção de dependências solucionáveis por Forwarding (\texttt{src\/ForwardingUnit.v}), e um outro para a detecção de dependências mais severas, que forçassem um \textit{stall} do pipeline para executar corretamente (\texttt{src\/HazardDetectionUnit.v}).

\subsection{Hazard Detection e Forwarding}

A unidade de \textit{forwarding} foi a primeira a ser implementada, sendo responsável por detectar dependências de dados entre a instrução no estágio ID/EX e resultados ainda não escritos no banco de registradores, gerando os sinais forwardA e forwardB para selecionar de onde cada operando da ULA deve vir. O módulo compara rs1 e rs2 com os registradores de destino presentes nos estágios EX/MEM e MEM/WB e, quando encontra correspondência válida, escolhe entre não fazer \textit{forwarding}, encaminhar diretamente do estágio de memória, ou encaminhar resultados que vêm do estágio de \textit{write-back}, diferenciando inclusive valores produzidos por operações da ULA e por instruções de carga (lw). Com isso, ele reduz a necessidade de \textit{stalls} e ajuda a manter o fluxo do pipeline mais eficiente.

Mesmo após a implementação da unidade de \textit{forwarding}, ainda não haviamos obtido uma execução correta, pois a instrução BEQ ainda gerava um "atraso" grande demais. A solução foi simples, com a inclusão da unidade de detecção de \textit{hazards}. Verifica se a instrução no registrador EXMEM é uma instrução de \textit{load}, e caso seja, gerar \textit{stall} se a próxima instrução precisar de usar o resultado do \textit{Load}.

Os resultados apresentados evidenciam o correto funcionamento do pipeline com os mecanismos de \textit{forwarding} e detecção de \textit{hazards} implementados. Mesmo diante de um programa com múltiplas dependências de dados e controle, o processador foi capaz de executar todas as instruções sem erros, como indicado pela validação final bem-sucedida. Observa-se ainda que o número reduzido de \textit{stalls} (apenas um) e a presença significativa de \textit{bypasses} demonstram a efetividade das técnicas adotadas para minimizar interrupções no fluxo do pipeline. O valor de CPI obtido permanece dentro do esperado para um cenário com hazards, confirmando a eficiência da solução proposta

\section{Conclusão}

Por meio deste relatório, documentamos nosso sucesso em adicionar a instrução de desvio condicional, bem como realizar os ajustes e implementação dos recursos necessários para dar capacidade ao processador de executar a pipeline corretamente, sem a necessidade de intervenção externa. Os resultados evidenciam uma baixa taxa de stalls e uma alta taxa de bypasses, com um CPI satisfatório para um cenário com diversos hazards, validando as alterações feitas.

Como trabalhos futuros, evidenciamos a relevância do suporte à mais extensões do RISC-V, que contribuem de forma positiva com o ecossistema da ISA. Além disso, a implementação de caches de múltiplos níveis e separadas entre cache de instrução e dados é um ponto importante para aprimorar as capacidades do processador e ainda lidar de forma eficiente com dependências de dados. Por fim, a implementação de uma pipeline superescalar, com suporte a execução fora de ordem, é uma adição muito importante para equiparar a arquitetura com outras atuais.
